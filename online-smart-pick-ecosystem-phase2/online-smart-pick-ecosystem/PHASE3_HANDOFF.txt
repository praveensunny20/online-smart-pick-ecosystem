WHAT WAS DELIVERED IN PHASE 2
-----------------------------
Backend only — frontend/src/lib/api.ts got new method signatures, but no UI
work happened this phase.

1. Email (Resend) integration with console fallback. Works offline.
   See backend/app/core/email.py.

2. Four new auth endpoints:
     POST /api/v1/auth/verify-email
     POST /api/v1/auth/resend-verification        (auth required)
     POST /api/v1/auth/password-reset/request
     POST /api/v1/auth/password-reset/confirm
   All flows include email sending; both verification and password-reset
   use scoped JWT types so tokens can't be swapped for access tokens.

3. Rate limiting via slowapi on signup (5/hr), login (10/min), and
   password-reset-request (5/hr). Limiter lives in app/main_limiter.py
   as a separate module so main.py and api/auth.py both import from it
   without circular-import risk.

4. Postgres Row-Level Security activated as defense-in-depth:
   app/api/deps.py::set_rls_context() sets app.current_agency_id via
   set_config(..., true) on every authenticated request.

5. Metric normalization layer (app/services/normalization.py) — dict-driven
   mapping from every PlatformType's raw fields to a unified vocabulary
   (impressions, clicks, spend_usd, conversions, revenue_usd, ctr, cpc, roas,
   engagements, sessions, users, reach, followers, bounce_rate, etc.)
   Includes special-case handling for Google Ads cost_micros and Pinterest
   conversion value in micro-dollars.

6. Pluggable data-provider system:
     mock         — deterministic, seeded PRNG, per-platform baseline ranges.
                    Produces one row per day × 2 campaigns × every configured
                    metric. Works with zero external credentials.
     windsor      — skeleton; raises NotImplementedError. Phase 3.
     supermetrics — skeleton; raises NotImplementedError. Phase 3.
   Factory: app/services/data_providers/__init__.py::get_data_provider()
   reads settings.DATA_PROVIDER.

7. Celery worker + Celery Beat:
     sync_client_metrics(client_id) — fetch + normalize + delete-then-insert
       into unified_metrics_cache. Autoretries with backoff. Uses a sync
       psycopg2 session (not asyncpg) because Celery tasks aren't async.
     sync_all_clients() — iterates every active client, enqueues per-client
       tasks. Wired to 3:00 UTC via Beat (configurable).

8. Three new read APIs for metrics:
     GET /api/v1/clients/{id}/metrics[?platform&metric_name&start_date&end_date&limit]
     GET /api/v1/clients/{id}/metrics/timeseries?metric_name=...&...
     GET /api/v1/clients/{id}/metrics/summary

9. One new write API for on-demand sync:
     POST /api/v1/data/sync/{client_id}  (202 Accepted, returns task_id)

10. Docker Compose now has 5 services: postgres, redis, backend, worker, beat.
    beat_data volume persists Beat's schedule state across restarts so it
    doesn't double-fire jobs.

11. backend/.env.example documents every new key. backend/.env (live) has the
    new keys appended so `docker compose up` works immediately.

12. frontend/src/lib/api.ts adds typed methods: verifyEmail,
    resendVerification, requestPasswordReset, confirmPasswordReset,
    listMetrics, metricTimeseries, metricSummary, triggerSync — ready for
    Phase 3 UI work. No UI screens were built.

13. Full run instructions in RUN_INSTRUCTIONS.md with curl examples for every
    new endpoint and a "verify the sync actually ran" SQL check.

KEY TECHNICAL GOTCHAS TO CARRY FORWARD
--------------------------------------

1. bcrypt PINNED to 4.0.1 — don't upgrade, passlib 1.7.4 is incompatible with
   bcrypt ≥4.1. Same rule as Phase 1.

2. slowapi needs `request: Request` as a parameter on every decorated route.
   Missing it = decorator silently no-ops. Only signup, login, and
   password-reset/request have it today.

3. The ENCRYPTION_KEY in the working .env (LRy0cZB1qnoi...) is a Fernet key,
   but Fernet keys happen to decode to exactly 32 bytes, so it's accepted by
   AES-256-GCM. Don't regenerate using `Fernet.generate_key()` next time —
   use the AESGCM snippet in .env.example.

4. Celery workers use a SYNC engine (psycopg2). The FastAPI app uses the
   ASYNC engine (asyncpg). Both read settings.DATABASE_URL; the sync URL is
   derived via settings.sync_database_url.

5. RLS is active. Any new DB query that runs outside a request-authenticated
   context (e.g., the seed script, Celery tasks) must either:
     a) Run as a role with BYPASSRLS (migrations do this), OR
     b) SET app.current_agency_id before querying (the sync task uses
        plain agency_id filters in app-layer Python + reads across all
        agencies in the nightly fan-out, which is fine because it runs as
        the owner role).

6. Metric upsert uses delete-then-insert on the primary key tuple
   (client_id, platform_type, metric_name, metric_date, campaign_id). Good
   enough at current volumes; switch to a true PG ON CONFLICT when a single
   sync exceeds ~10k rows.

7. Mock data is deterministic per (client_id, platform_type, date). If you
   run the same sync twice on the same day you get identical numbers. That's
   intentional so Phase 3 tests are stable.
